## Felix Protocol Analysis

**What it is:** One of three protocols on Hyperliquid's HIP-3 framework for tokenized equities (alongside xyz/Trade.xyz and vnti/Ventuals).

**Key characteristics:**
- Built by RedStone (oracle provider)
- Uses HyperStone oracle (3-second price updates)
- Assets: CRCL, TSLA, COIN, NVDA (up to 3x leverage)
- Settlement in USDH (Hyperliquid's native stablecoin - liquidity issue)
- Has haltTrading mechanism for emergency position closure
- **Daily volume:** Low single-digit millions (much smaller than Trade.xyz's $65M+ for NVDA)

**Risks identified:**
1. USDH liquidity trap (not USDC - exit risk if USDH depegs)
2. Thin order books ($50k order can move price 2-5%)
3. Emergency halt risk (positions force-closed at settlement price)
4. Oracle lag (3s updates can miss fast-moving markets like NVDA earnings)

**Verdict:** Not for serious NVDA exposure. Better alternatives: Trade.xyz (same ecosystem, better liquidity) or centralized (Bitget/Bybit). Felix is for experimental assets and arbitraging USDH ecosystem.

## Dominant Presence in Trading - Strategic Framework

**Core insight:** The dominant presence won't be a venue—it'll be infrastructure that abstracts venue complexity away. Intent-based execution layers that route across CEX, DEX, on-chain, off-chain.

**Five Forces Reshaping Trading:**
1. **Great Convergence** - TradFi/DeFi merge; all assets as tokens in one interface
2. **Agentic Execution** - Human states intent, agent handles venue selection, execution, risk management
3. **Liquidity Fragmentation** - More venues = better price discovery but worse execution; aggregation layers win
4. **Custody Spectrum** - Flexible tiers (self-custody → smart contract → exchange → TradFi)
5. **Speed vs. Trust** - Different winners for different user types

**Prediction by timeline:**
- 2-3 years: Hyperliquid/Bybit (active), Coinbase (retail)
- 5 years: Best agent execution layer (incumbent or new entrant)
- 10 years: Invisible - trading happens inside AI assistants, venue abstracted

## ATLAS by General Intelligence Capital - Analysis

**What it is:** Open-source AI trading framework with agent swarms, evolutionary prompt optimization, and reflexivity engine. 944 stars on GitHub.

**Architecture (25+ agents across 4 layers):**
- Layer 1: Macro (10 agents) - set regime
- Layer 2: Sector desks (7 agents) - industry dynamics
- Layer 3: Superinvestors (4 agents) - Druckenmiller, Aschenbrenner, Baker, Ackman styles
- Layer 4: Decision (4 agents) - CRO (adversarial), Alpha Discovery, Execution, CIO

**Key innovations:**
1. **"Prompts as weights"** - Optimize prompts via market feedback (Sharpe = loss function) instead of GPU training; $20 VM replaces H100
2. **Autoresearch loop** - Worst agent gets prompt modified, 5-day trial, commit if Sharpe improves, revert if not
3. **PRISM (regime-specific training)** - Separate cohorts for bull, crisis, rate tightening, recovery, euphoria; agents evolve different "instincts"
4. **Reflexivity engine** - Model feedback loops (price→fundamentals, P&L→behavior, narrative→flows); Soros in code
5. **MiroFish integration** - Train on simulated futures, not just past data

**Backtest claims:**
- 18-month period: +22% in 173 days live deployment
- Best pick: AVGO at $152, held for +128%
- But: Three regimes had negative returns (-13%, -30%, -29%)

**Critical concerns:**
1. Trained prompts are proprietary (open-source framework is incomplete)
2. Autoresearch too slow for crisis regimes (COVID: "too fast for autoresearch")
3. Survivorship bias in backtests (what's max drawdown?)
4. Darwinian selection risks herding
5. MiroFish dependency (student project, unproven at scale)

**Strategic connection to user's work:**
ATLAS is exactly the kind of system that would consume APM infrastructure:
- Agent consensus feeds
- Cross-venue surveillance
- Reflexive detection
- Conditional execution

**Verdict:** Clever architecture, validates user thesis, but durability unproven. Worth studying closely; worth partnering with (reach out to Chris Worsey at GIC).
